# リサーチ: カート管理

**作成日**: 2026-02-08
**ブランチ**: `002-cart-management`

## 技術的な決定事項

### 1. カートの永続化方式

- **決定**: インメモリストア（`globalThis.__cartStore`）を使用
- **根拠**: 既存の `src/infrastructure/repositories/cart.ts` がこのパターンで実装済み。仕様で「インメモリとし、DB永続化は将来フェーズ」と明記。カタログ機能（001）と同じパターンを踏襲
- **検討した代替案**: localStorage（クライアント側永続化） → サーバーサイドでの在庫検証ができないため不採用

### 2. カート識別方式

- **決定**: セッションベースの userId でカートを識別。認証なし（ゲスト）でもカート利用可能
- **根拠**: 既存のサンプル実装（`src/samples/domains/cart/api/usecases.ts`）が buyer ロール認証を要求しているが、仕様書では「認証不要」。ゲストユーザー向けにはセッションIDまたは仮ユーザーIDで対応
- **検討した代替案**: Cookie ベースの匿名ID → 既存のセッション管理と整合性がとれるため、認証済みユーザーの userId を使用し、未認証時はセッション経由で仮IDを割り当てる

### 3. 消費税計算ロジック

- **決定**: 商品合計（全アイテムの小計の合算）に対して10%を乗じ、`Math.floor()` で端数切り捨て
- **根拠**: 仕様書 FR-017「商品合計に対して計算し端数を切り捨て」に準拠。商品単位での端数処理は行わない
- **検討した代替案**: 商品単位で消費税を計算 → 合計金額に誤差が生じるため不採用

### 4. カート更新のUI同期方式

- **決定**: `cart-updated` カスタムイベント（`src/templates/ui/utils/events.ts` で定義済み）を使用
- **根拠**: 既存の `src/app/(buyer)/layout.tsx` が `cart-updated` イベントを監視してヘッダーのカート件数を更新する仕組みが実装済み
- **検討した代替案**: React Context / Zustand → 既存イベント機構で十分であり、追加依存を避ける

### 5. 二重送信防止

- **決定**: ボタンの disabled 状態管理（API呼び出し中は無効化）
- **根拠**: シンプルで確実。サーバー側の冪等性は同一商品追加時の数量加算で自然に担保される
- **検討した代替案**: サーバー側トークン方式 → インメモリストアでは不要な複雑さ
