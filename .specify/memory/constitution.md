# トレーニングECサイト 憲法

## コア原則

### I. テンプレート駆動開発

- UI・API・テストの実装は `src/templates/` 配下のテンプレートを基に行わなければならない（MUST）
- 共通基盤機能（認証・エラーハンドリング・バリデーション・ログ）は `src/foundation/` を使用しなければならない（MUST）
- テンプレートから逸脱する場合は、その理由を明示し、レビューで承認を得なければならない（MUST）

**根拠**: テンプレートに基づく統一的な実装により、コードの一貫性と品質を維持し、開発速度を向上させる。

### II. ドメイン分離

- 各ドメインは `src/domains/[domain]/` に独立して実装しなければならない（MUST）
- ドメイン内部は `types/`、`api/`、`ui/` サブディレクトリに分離しなければならない（MUST）
- ドメイン間の依存は API 経由のみとし、直接的なモジュール参照を禁止する（MUST NOT）
- `src/contracts/` の共有インターフェースに準拠し、`src/samples/` を参考に実装する（MUST）

**根拠**: ドメイン境界を明確にすることで、各ドメインの独立した開発・テスト・デプロイを可能にし、変更の影響範囲を最小化する。

### III. テストファースト（非交渉）

- TDD（Red → Green → Refactor）サイクルを厳守しなければならない（MUST）
  1. テストを先に記述する
  2. テストが失敗することを確認する（Red）
  3. テストが通る最小限の実装を行う（Green）
  4. リファクタリングを行う（Refactor）
- Red フェーズの確認をスキップしてはならない（MUST NOT）
- 各ユースケースにテストタスクを含めなければならない（MUST）
- テストカバレッジ80%以上を維持しなければならない（MUST）

**根拠**: テストファーストにより、仕様の正確な理解を保証し、リグレッションを防止する。Red フェーズの確認は、テストが正しく失敗することを保証する不可欠なステップである。

### IV. 実装ワークフロー

- `src/domains/` の暫定スキャフォールドを本番実装に置き換える形で開発する（MUST）
- `src/contracts/` に準拠し、`src/samples/` を参考にして実装しなければならない（MUST）
- ユーザーストーリー単位でフェーズを分割し、各ストーリーを独立して実装・テスト可能にしなければならない（MUST）
- ドメイン実装時にレイアウトファイルの navLinks にナビゲーションリンクを追加しなければならない（MUST）
  - 購入者向け: `src/app/(buyer)/layout.tsx` の navLinks に追加（例: `{ href: '/catalog', label: '商品一覧' }`）
  - 管理者向け: `src/app/admin/layout.tsx` の navLinks に追加（例: `{ href: '/admin/products', label: '商品管理' }`）
  - navLinks はデフォルトで空（購入者）またはダッシュボードのみ（管理者）。実装したドメインのリンクのみ追加する

**根拠**: 段階的な置換アプローチにより、既存のサンプル実装を参考にしながら安全に本番コードへ移行できる。ナビゲーション追加をワークフローに含めることで、実装漏れを防止する。

## 技術スタック・品質基準

### 技術スタック

| カテゴリ | 技術 |
|---------|------|
| フレームワーク | Next.js 14 (App Router) |
| 言語 | TypeScript 5 (strict mode) |
| UIライブラリ | React 18 |
| スタイリング | Tailwind CSS 3 |
| バリデーション | Zod |
| 単体・統合テスト | Vitest |
| E2Eテスト | Playwright |

### 品質基準

- TypeScript: strict モード有効、コンパイルエラー0件（MUST）
- ESLint: エラー0件（MUST）
- テストカバレッジ: 80%以上（MUST）
- E2Eテスト: 主要ユーザー導線をカバー（MUST）
- パフォーマンス: 一覧ページ初回ロード3秒以内（SHOULD）

### 認証・認可

- ロール: buyer（購入者）、admin（管理者）
- セッション管理: Cookie ベース
- 認可方式: RBAC（Role-Based Access Control）

## 命名規約・認証認可

### 命名規約

| 対象 | 規約 | 例 |
|------|------|----|
| ファイル名 | kebab-case | `product-list.tsx` |
| コンポーネント | PascalCase | `ProductList` |
| 関数 | camelCase | `getProducts` |
| 定数 | UPPER_SNAKE_CASE | `MAX_ITEMS` |
| 型 | PascalCase | `ProductType` |

### 言語方針

- すべての仕様書・計画・タスク・実装に関するドキュメントは日本語で記述しなければならない（MUST）

## ガバナンス

- この憲法はプロジェクトの全開発プラクティスに優先する（MUST）
- 憲法の修正には以下を要する:
  1. 修正内容の文書化
  2. 影響範囲の分析（依存テンプレート・既存実装への影響）
  3. マイグレーション計画の策定（既存コードへの適用方針）
- すべてのプルリクエスト・レビューで憲法準拠を検証しなければならない（MUST）
- 複雑さの追加は正当化が必要であり、よりシンプルな代替案が不十分である理由を明記しなければならない（MUST）
- バージョニング方針:
  - MAJOR: 原則の削除・再定義等の後方互換性のない変更
  - MINOR: 新規原則・セクションの追加、既存ガイダンスの大幅な拡充
